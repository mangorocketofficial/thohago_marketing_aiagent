# Phase 5-3: Full Campaign Plan Generation (4-Step Chain)

- Date: 2026-03-04
- Scope: Campaign Plan AI Chain + Markdown Assembly + Storage
- Status: Draft (Updated)
- Depends on: 5-0 (Skill Router), 5-2 (Enriched RAG Context)

## 1) Context

Current `generateCampaignPlan` returns a legacy 6-field skeleton (`objective`, `channels`, `duration_days`, `post_count`, `content_types`, `suggested_schedule`).

Phase 5-3 upgrades this to a production-ready campaign plan generated by a 4-step AI chain.

Design principle: structured data first, markdown last.
- AI steps generate typed JSON only.
- Final markdown is assembled by deterministic template code.
- This keeps format stable and enables section-level revisions.

## 2) Key Design Updates (Accepted)

1. Encoding safety
- This document and all generated templates/prompts must use UTF-8.
- No mojibake text is allowed in prompts, placeholders, or section titles.

2. Context injection policy (cost and latency optimized)
- Step A: Full enriched RAG context.
- Step B: Compact fact pack only (summary of brand constraints + critical facts).
- Step C and Step D: Prior step outputs + micro fact pack (minimal context), not full RAG repeat.
- Reason: vector retrieval is cheap, but LLM prompt tokens are still billed and increase latency when repeated.

3. Dependency-safe failure handling
- Replace "continue with null and generate downstream anyway" with dependency-aware blocking.
- If prerequisite step fails, dependent steps are marked `blocked_by_dependency` and not generated.

4. Typed JSON guarantee
- Every step output must pass runtime schema validation.
- On parse/schema failure: 1 repair retry with strict JSON-only prompt.
- If still invalid: mark step `failed` and include structured error metadata.

5. SLA and observability
- SLA split:
  - Target: <= 30s
  - Hard timeout: 45s
- Collect per-step metrics: prompt tokens, completion tokens, latency_ms, retry_count.

6. Revision concurrency control
- Section-level revision must use optimistic concurrency (`expected_updated_at` or equivalent).
- If stale update is detected, return 409 conflict and require client refresh/retry.

## 3) Chain Architecture

```text
User message + enriched RAG context (Phase 5-2)
  -> Step A: Audience & Messaging
     Input: user message + full RAG
     Output: AudienceMessagingData
  -> Step B: Channel Strategy
     Input: Step A + compact fact pack
     Output: ChannelStrategyData
  -> Step C: Content Calendar
     Input: Step A + Step B + micro fact pack
     Output: ContentCalendarData
  -> Step D: Assets/KPIs/Risks/Next Steps
     Input: Step A + Step B + Step C + micro fact pack
     Output: ExecutionData
  -> Markdown Assembler (no AI)
     Input: A + B + C + D
     Output: 10-section markdown
  -> Store in campaigns.plan_chain_data + campaigns.plan_document
```

## 6) Failure and Dependency Rules

Dependency graph:
- Step A: independent
- Step B: depends on A
- Step C: depends on A and B
- Step D: depends on A, B, C

Rules:
- If A fails -> B/C/D = blocked_by_dependency
- If B fails -> C/D = blocked_by_dependency
- If C fails -> D = blocked_by_dependency
- If D fails -> only D failed
- No downstream generation when required upstream data is missing

## 7) JSON Validation and Retry Policy

Per step:
1. Parse JSON object from model response.
2. Validate runtime schema.
3. If parse/schema fails, retry once with repair prompt:
   - include previous raw output
   - require strict JSON only
   - require same schema
4. If still invalid:
   - store `null` result for that step
   - mark step meta as `failed`
   - continue only where dependency graph allows

## 8) Token and Timeout Budget

- Output target (guideline):
  - Step A: <= 1200
  - Step B: <= 900
  - Step C: <= 1200
  - Step D: <= 1200
- Total timeout:
  - Target: 30s
  - Hard limit: 45s

Implementation note:
- Full RAG repeated across all steps is prohibited by policy.
- B/C/D receive only compact inputs from A + constrained fact packs.

## 10) Section-Level Revision

User requests section revision -> map to step and cascade rerun from that step to D.
- Revise A -> rerun A/B/C/D
- Revise B -> rerun B/C/D
- Revise C -> rerun C/D
- Revise D -> rerun D

Concurrency:
- Revision request must include `expected_updated_at` (or equivalent version token).
- On mismatch, return 409 conflict.
