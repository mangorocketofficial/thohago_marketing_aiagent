# Phase 7-2.2: Template Schema Redesign — Render Lock + Optional Meta

- Date: 2026-03-05
- Status: In Progress
- Scope: Lock rendering schema to photos/texts only, while allowing non-rendering optional `meta`.
- Depends on: Phase 7-2a, Phase 7-2.1
- Breaks: Internal template JSON shape (`overlays/badge/header` -> flattened photo/text slots)

---

## 1) Problem

Template schema kept expanding per design style (`badge`, `header`, `decorations`, `z_index`, etc.).
This caused renderer branching, editor coupling to decoration types, and higher onboarding cost for new templates.

---

## 2) Design Principle

**Sharp renders only photos and text. Everything else is baked into `background.png`.**

- Decorative assets (logos, ribbons, shadows, bars, icons, frames) are pre-baked by design.
- Runtime compose does only: `background + photos + text SVGs`.
- Rendering contract stays fixed as styles evolve.

---

## 3) Final Schema (Strict Render Fields + Optional Meta)

```typescript
interface TemplateConfig {
  template_id: string;
  template_name: string;
  size: { width: number; height: number };
  photos: TemplatePhotoSlot[];
  texts: TemplateTextSlot[];
  meta?: Record<string, unknown>; // non-rendering only
}

interface TemplatePhotoSlot {
  id: string;
  label: string;
  x: number;
  y: number;
  width: number;
  height: number;
  fit: "cover" | "contain";
  optional?: boolean;
}

interface TemplateTextSlot {
  id: string;
  label: string;
  x: number;
  y: number;
  width: number;
  height: number;
  font_size: number;
  font_color: string;
  font_weight?: "normal" | "bold";
  align: "left" | "center" | "right";
}
```

### Rendering Contract Rules

- Allowed rendering fields: only `size`, `photos[]`, `texts[]`.
- `meta` must be non-rendering metadata (description/thumbnail/tags, etc.).
- No runtime rendering behavior may depend on `meta`.

---

## 4) Removed Rendering Fields

| Removed field | Handling |
|---|---|
| `overlays` | Flattened into top-level `photos` / `texts` |
| `badge` | Badge shape baked in PNG, text moved to `texts[]` |
| `header` | Baked in PNG |
| `z_index` | Layer order fixed by compose order |
| `font_style` | Removed from render schema (Pretendard only) |
| `bg_shape`, `bg_color`, `text_shadow`, `decorations`, `browser_window`, `page_indicator`, `opacity`, `style` | Baked in PNG |

---

## 5) Asset Structure

Each template folder should include:

```
assets/templates/{template_id}/
  background.png
  config.json
  thumbnail.png
```

`config.json` uses the schema above (`meta` optional).

---

## 6) Composer Contract

Target compose contract:

```typescript
composeInstagramImage({
  templateId,
  userImages,
  overlayTexts,
  outputFormat
})
```

- `templateId` resolves `config.json + background.png` through registry.
- Required image count is based on `photos.filter(slot => !slot.optional).length`.
- No background branching (solid/gradient/etc.) in runtime.

1. `@repo/media-engine` schema/registry/composer switched to `photos/texts + optional meta`.
2. `koica_cover_01` preset config migrated to flattened schema.
3. API template response and metadata validation paths switched to `template.photos/template.texts`.

---

## 8) Acceptance Criteria

1. Renderer reads only `size/photos/texts` for composition.
2. `meta` is optional and ignored by renderer.
3. No runtime usage of `badge/header/overlays/z_index/font_style` in compose path.
4. `pnpm --filter @repo/media-engine type-check` passes.
5. `pnpm --filter @repo/api type-check` and `pnpm --filter @repo/desktop type-check` pass.
6. Editor overlay zones render dynamically from `texts[]`.

---

## 9) Decisions

**Why not hard lock to exactly 5 keys:**
A strict 5-key lock blocks useful future metadata. We lock rendering fields and allow optional non-rendering `meta`.

**Why keep width/height (not w/h):**
Using existing naming avoids migration noise and keeps code readability.

**Why keep templateId-based compose API:**
Template lookup, caching, and validation stay centralized in registry; callers avoid direct file-path coupling.
