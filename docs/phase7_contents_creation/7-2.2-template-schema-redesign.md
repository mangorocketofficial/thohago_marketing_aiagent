# Phase 7-2.2: Template Schema Redesign — 5-Field Lock + Background Bake

- Date: 2026-03-05
- Status: Planning
- Scope: Reduce template config schema to 5 fixed fields. All visual decoration baked into background.png. Sharp renders photos and text only.
- Depends on: Phase 7-2a (current template system), Phase 7-2.1 (local compose refactor)
- Breaks: Template JSON schema (internal, not user-facing)

---

## 1) Problem

The current template config schema grows with every new template style:

- Template 1 (KOICA airmail) added: `badge`, `header`, `bg_shape`, `bg_color`, `font_style`, `z_index`
- Template 2 (ODA retro pixel) added: `decorations[]`, `browser_window`, `page_indicator`, `text_shadow`, `opacity`, `style`

Each new template design introduces new schema fields. This creates:

1. **Unbounded schema growth** — every visual effect becomes a new field.
2. **Sharp renderer complexity** — must handle circles, gradients, shadows, browser frames, pixel art decorations.
3. **Editor complexity** — must know about every decoration type to position overlay zones correctly.
4. **Template onboarding friction** — creating a new template requires understanding the full schema.

---

## 2) Design Principle

**Sharp renders only two things: photos and text. Everything else is baked into background.png.**

This means:
- Colored bars, badges, shadows, decorations, frames, icons, logos → drawn into background.png by the designer.
- Sharp loads background.png, places user photos at coordinates, places SVG text at coordinates. Done.
- Schema never grows regardless of visual complexity.

---

## 3) Final Schema (5 Fields — No Additions Allowed)

```typescript
interface TemplateConfig {
  template_id: string;
  template_name: string;
  size: { width: number; height: number };
  photos: TemplatePhotoSlot[];
  texts: TemplateTextSlot[];
}

interface TemplatePhotoSlot {
  id: string;
  label: string;
  x: number;
  y: number;
  w: number;
  h: number;
  fit: "cover" | "contain";
  optional?: boolean;
}

interface TemplateTextSlot {
  id: string;
  label: string;
  x: number;
  y: number;
  w: number;
  h: number;
  font_size: number;
  font_color: string;
  font_weight?: "normal" | "bold";
  align: "left" | "center" | "right";
}
```

### Removed fields (baked into background.png)

| Removed Field | Reason |
|---|---|
| `badge` | Badge circle/ribbon drawn in PNG. Badge text moved to `texts[]`. |
| `header` | Logos and branding are fixed per template. Drawn in PNG. |
| `overlays` wrapper | Flattened. `photos` and `texts` are top-level arrays. |
| `bg_shape` / `bg_color` | Colored bars, circles behind text drawn in PNG. |
| `text_shadow` | Shadow effect drawn in PNG behind text slot position. |
| `font_style` | Removed. All text uses Pretendard. Bold/normal via `font_weight`. |
| `decorations[]` | Pixel hearts, cursors, chicks, glasses, hourglasses → PNG. |
| `browser_window` | Browser frame UI → PNG. |
| `page_indicator` | Carousel dots → PNG. |
| `style` | Visual style tag unnecessary. PNG *is* the style. |
| `z_index` | Removed. Composite order = array index order. |
| `opacity` | Removed. Background blur/dim effects pre-baked in PNG. |
| `example_text` | Removed from schema. Documented in README only. |
| `description` | Removed from schema. Documented in README only. |

## 4) Template Asset Structure

Each template is a folder with exactly 3 files:

```
assets/templates/
  {template_id}/
    background.png    ← 1080x1080, all decorations baked in
    config.json       ← 5 fields only
    thumbnail.png     ← 200x200 preview for template picker
```

No other files. No sub-folders. No composite_example.js. No README per template.

## 7) Sharp Composer Changes

### Before (current image-composer.ts)

```
resolveBackground(config)     → solid / gradient / image branching
applyDarkOverlay(buffer)      → semi-transparent overlay
preprocessUserImage(path)     → resize/crop single image
buildTextOverlaySvg(main, sub) → fixed 2-slot text rendering
composeImage(layers)          → final composite
```

### After

```
loadBackground(templateDir)                          → sharp(background.png)
preprocessPhotos(config.photos, imagePaths)          → resize each photo to slot dimensions
buildTextSvgs(config.texts, values: Record<id, str>) → SVG buffer per text slot
compose([background, ...photos, ...textSvgs])        → composite in array order
```

### Removed functions

| Function | Reason |
|---|---|
| `resolveBackground()` | Background is always a PNG file. No branching. |
| `createSolidBackground()` | Solid colors baked into PNG. |
| `createGradientBackground()` | Gradients baked into PNG. |
| `applyDarkOverlay()` | Dark overlays baked into PNG. |
| `applyRoundedCorners()` | Corner rounding baked into PNG frame design. |
| `applyImageOpacity()` | Opacity effects baked into PNG. |

### Simplified compose pipeline

```typescript
async function composeInstagramImage(
  templateDir: string,
  photoInputs: Record<string, string>,   // { "main_photo": "/path/to/img.jpg" }
  textValues: Record<string, string>,     // { "title": "Hello", "author": "Kim" }
): Promise<{ buffer: Buffer; width: number; height: number }> {
  const config = loadConfig(templateDir);
  const bgPath = path.join(templateDir, "background.png");

  const layers: sharp.OverlayOptions[] = [];

  // 1. Photos
  for (const slot of config.photos) {
    const inputPath = photoInputs[slot.id];
    if (!inputPath && slot.optional) continue;
    if (!inputPath) throw new Error(`Missing required photo: ${slot.id}`);

    const buf = await sharp(inputPath)
      .resize(slot.w, slot.h, { fit: slot.fit, position: "centre" })
      .toBuffer();

    layers.push({ input: buf, left: slot.x, top: slot.y });
  }

  // 2. Texts
  for (const slot of config.texts) {
    const text = textValues[slot.id];
    if (!text) continue;

    const svg = buildSingleTextSvg(slot, text);
    layers.push({ input: svg, left: slot.x, top: slot.y });
  }

  // 3. Compose
  const result = await sharp(bgPath)
    .composite(layers)
    .png()
    .toBuffer();

  return { buffer: result, width: config.size.width, height: config.size.height };
}
```

## 10) Acceptance Criteria

1. Every template folder contains exactly 3 files: `background.png`, `config.json`, `thumbnail.png`.
2. Every `config.json` has exactly 5 top-level fields: `template_id`, `template_name`, `size`, `photos`, `texts`.
3. No config.json contains: `badge`, `header`, `decorations`, `browser_window`, `page_indicator`, `bg_shape`, `bg_color`, `text_shadow`, `font_style`, `style`, `z_index`, `opacity`, `example_text`, `description`.
4. `grep -r "bg_shape\|bg_color\|text_shadow\|decorations\|browser_window\|page_indicator" assets/templates/` returns zero matches.
5. Sharp compose pipeline has no branching for background type (solid/gradient/image). Always loads `background.png`.
6. `composeInstagramImage()` accepts `(templateDir, photoInputs, textValues)` — no other visual parameters.
7. Composed output is visually identical to current composite test images (decorations visible from PNG, text and photos at correct positions).
8. `pnpm --filter @repo/media-engine type-check` passes with new schema types.
9. Editor renders dynamic text overlay zones from `texts[]` array (not fixed `mainText/subText`).

## 11) Decisions

**Why 5 fields and not 8:**
Allowing "just a few more" fields (badge, header, bg_color) sets a precedent. Template 2 added 6 new fields. Template 3 would add more. The only sustainable boundary is a hard cap: Sharp draws photos and text. PNG handles everything else. No exceptions.

**Why bake text backgrounds (colored bars, circles) into PNG:**
Alternative: add `bg_color`/`bg_shape` to `TemplateTextSlot` so Sharp draws the bar behind text. This seems small but requires SVG rect/circle rendering logic, and every new shape (rounded rect, pill, diamond) becomes a schema extension. Baking into PNG keeps Sharp purely compositional.

**Why remove `text_shadow`:**
Sharp SVG rendering has limited shadow support. Baking a shadow shape into the background at the exact text position is more reliable and visually consistent. If the text position shifts during editing, the shadow in PNG won't match — but this is acceptable because text positions are template-fixed, not user-adjustable.

**Why remove `z_index`:**
Sharp `composite()` applies layers in array order. First item = bottom, last = top. Array index *is* the z-order. An explicit z_index field adds complexity without benefit.

**Why `photos` can be empty array:**
Some templates are text-only (e.g., NGO job list, CTA cards). `photos: []` is valid. The compose pipeline simply skips the photo loop.
