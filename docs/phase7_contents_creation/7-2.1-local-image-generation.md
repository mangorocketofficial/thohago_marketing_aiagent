# Phase 7-2.1: Instagram Local Image Composition Refactoring

- Date: 2026-03-05
- Status: Planning
- Scope: Move Instagram compose/preview from API signed URL path to Electron local filesystem path.
- Depends on: Phase 7-2a (template/sharp engine), Phase 7-2b (canvas editor)
- Compatibility: `recompose/signed-url` contracts are removed in final cutover (internal API/IPC breaking change).

---

## 1) Problem

Current flow composes on API and stores preview artifacts in private Supabase Storage.  
For desktop-first editing this causes avoidable complexity:

1. Signed URL lifecycle management (TTL refresh, expiration recovery, retries).
2. Edit round-trip latency (renderer -> main -> API -> storage -> signed URL -> renderer).
3. Repeated bandwidth/storage cost for intermediate artifacts.
4. Edit path hard-depends on network even when source files are already local.

---

## 2) Goals

1. Compose in Electron main process using shared media engine.
2. Cache composed output at `{watchPath}/contents/.instagram-cache/{contentId}/composed.png`.
3. Remove signed URL lifecycle from Instagram editor path.
4. Keep UX parity (template switch, overlay edits, image replace, debounced recomposition).
5. Keep LLM generation and image-selection intelligence on API.
6. Persist visual edits (`template_id`, `overlay_*`, `image_file_ids`) to DB via dedicated metadata save contract.

---

## 3) Architecture (Target)

### Before (Server compose)

```
Renderer -> IPC -> Main -> HTTP -> API
API -> Sharp compose -> Supabase upload -> signed URL
Main -> Renderer (signed URL + expiresAt)
```

### After (Local compose)

```
Renderer -> IPC -> Main
Main -> local Sharp compose -> local cache write -> data URL thumbnail
Main -> Renderer (thumbnail + composedPath)
```

### API role after cutover

- Keep: LLM caption generation, template/image selection, content persistence.
- Remove from Instagram editor path: server recomposition endpoint and signed URL endpoint.
- Add/keep metadata update endpoint for visual state persistence.

---

## 4) Cutover Strategy (Mandatory)

This refactor must be shipped in 2 stages.

### Stage A (Compatibility Cutover)

1. Add local compose IPC (`content:compose-local`) and local-cache download path.
2. Add metadata save API/IPC for visual edits.
3. Extend chat generation metadata with local compose seed:
   - `template_id`, `overlay_main`, `overlay_sub`, `image_file_ids`
4. Keep existing server `recompose/signed-url` endpoint available during verification.

### Stage B (Final Removal)

1. Remove server `recompose/signed-url` routes.
2. Remove API-side signed URL / storage upload logic in Instagram generation path.
3. Remove renderer/main signed URL refresh code and IPC contracts.
4. Verify no Instagram path references remain:
   - `createSignedUrl`
   - `content-images-private`
   - `content:get-signed-url`
   - `content:recompose`

---

## 5) Design Decisions (Updated)

### D1. Breaking change is explicit

This is not "Breaks: None".  
Internal API/IPC contracts are changed in Stage B and require synchronized desktop+api deployment.

### D2. Visual metadata persistence is required

Local compose alone is insufficient.  
Without metadata persistence, template/overlay/image edits are lost across reload/session handoff.

### D3. Server compose policy is single-source

Final state: API does not compose preview artifacts for editor flow.  
Only local main process composes preview artifacts.

### D4. Chat card seed contract

Chat card local preview must not depend on signed URL.  
Generation metadata must include compose seed fields listed in Stage A.

### D5. Shared package build contract

`@repo/media-engine` must publish JS from `dist` (same convention as other workspace packages), not `src/index.ts` runtime entry.

### D6. Offline support scope

"Full offline editing" means:
- recomposition with already-known image ids/paths and existing local files.
- preview and download from local cache.

It does not include:
- new server-assisted image discovery/selection during network outage.
- server-driven regenerate flows during outage.

## 7) API Refactor

### 7.1 Generation path (`instagram-generation`)

- Remove storage upload + signed URL creation.
- Remove `composed_image_storage` metadata writes.
- Return `requires_local_compose: true`.
- Keep template/overlay/image seed in metadata and chat completion metadata.

### 7.2 Metadata persistence endpoint

Add endpoint (example):

- `PATCH /orgs/:orgId/contents/:contentId/instagram-metadata`

Body:

```json
{
  "template_id": "center-image-bottom-text",
  "overlay_main": "string",
  "overlay_sub": "string",
  "image_file_ids": ["..."],
  "expected_updated_at": "ISO optional"
}
```

Behavior:

- Validate channel/content_type are instagram/image.
- Enforce slot count against template image area count.
- Resolve file ids and persist `image_paths`.
- Return updated row + `updated_at`.

### 7.3 Route removal (Stage B)

Remove:

- `POST /orgs/:orgId/contents/:contentId/recompose`
- `GET /orgs/:orgId/contents/:contentId/signed-url`

## 8) Desktop/Electron Refactor

### 8.1 Main IPC

Add:

- `content:compose-local`
- `content:save-instagram-metadata`

Change:

- `content:download-image` reads from local cache path and copies to save target.

Remove in Stage B:

- `content:get-signed-url`
- `content:recompose`

### 8.2 Renderer runtime/types

Update preload + `global.d.ts`:

- add `composeLocal`
- add `saveInstagramMetadata`
- remove signed URL APIs in Stage B

### 8.3 Editor runtime

- Replace signed URL lifecycle hook with local compose hook.
- Keep debounce + latest-wins response guard.
- On visual edits: compose locally then persist metadata via new API/IPC.

### 8.4 Chat generation card

- Stop rendering `preview_url`.
- Compose local thumbnail using metadata seed fields.
