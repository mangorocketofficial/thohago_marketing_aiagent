## 1. 프로젝트 개요

### 비전
비영리/NGO/소셜벤처/사회적기업 도메인 특화 마케팅 AI 에이전트. 각 기관의 마케팅 주니어 직원을 대체하는 수준의 자율 마케팅 자동화 플랫폼.

### 핵심 가치
- 기관별 마케팅 보이스/전략 학습 및 고도화
- 워크플로우 자동화 (콘텐츠 생성 → 승인 → 발행)
- 도메인 특화 데이터 축적으로 시간이 지날수록 고도화
- 텔레그램 기반 자연어 인터페이스

### 타겟 사용자
비영리/NGO/소셜벤처/사회적기업의 마케팅 담당자 (1인 담당 or 소규모 팀)

## 4. 레이어 1 - 데이터

### 4.1 기술 스택
- **DB**: Supabase (PostgreSQL)
- **벡터 검색**: pgvector 익스텐션 (RAG용)
- **파일 스토리지**: Supabase Storage (이미지/영상)
- **보안**: Row Level Security (RLS) - 기관별 데이터 완전 분리
- **실시간**: Supabase Realtime (채팅 동기화)

### 4.2 멀티테넌트 구조

```
Supabase (중앙 DB)
├── 공통 테이블
│   ├── users                  ← 사용자 계정
│   ├── organizations          ← 기관 목록
│   └── subscriptions          ← 구독 정보
│
└── 기관별 분리 (org_id 기반 RLS)
    ├── org_brand_settings     ← 브랜드 보이스/금지목록
    ├── org_campaigns          ← 캠페인 정보
    ├── org_contents           ← 생성 콘텐츠 및 발행 히스토리
    ├── org_performance        ← 성과 데이터
    ├── org_chat_history       ← 에이전트 대화 내역
    └── org_rag_embeddings     ← pgvector RAG 임베딩
```

### 4.3 RAG 구성 데이터 (기관별)

```
RAG 문서 구조
├── 기초 정보
│   ├── 기관명, 설립연도, 미션/비전
│   ├── 주요 사업 분야, 활동 지역
│   ├── 타겟 오디언스
│   └── 주요 파트너/후원사
│
├── 마케팅 보이스
│   ├── 브랜드 톤 (따뜻한/전문적/긴박한)
│   ├── 자주 쓰는 표현/슬로건
│   ├── 선호 콘텐츠 스타일
│   └── 폰트/색상 브랜드 가이드
│
├── 금지 목록
│   ├── 절대 금지 단어/표현
│   ├── 민감 이슈 (정치적/종교적)
│   └── 과거 문제가 됐던 표현 사례
│
└── 이전 마케팅 데이터
    ├── 과거 발행 콘텐츠 전체
    ├── 성과 좋았던 콘텐츠 (가중치 높음)
    └── 성과 나빴던 콘텐츠 (반면교사)
```

### 4.4 memory.md 구조 (오케스트레이터용)

```markdown
# [기관명] 마케팅 메모리

## 장기 메모리 (안정적)
- 미션/비전
- 브랜드 보이스
- 금지 목록

## 단기 메모리 (자주 업데이트)
- 현재 진행 캠페인
- 최근 성과 요약
- 최근 이슈/공지

## 누적 인사이트
- 최적 발행 시간대
- 잘 된 CTA 문구 TOP5
- 채널별 성과 패턴
```

## 5. 레이어 2 - 서비스 (9개 모듈)

### 모듈 01: 온보딩 서비스
```
역할: 신규 기관 초기 셋업 및 RAG 초기 구축
입력: 기관 기초정보, 기존 마케팅 자료 업로드
출력: 완성된 RAG 임베딩 + memory.md 초안
의존성: db, rag
```

### 모듈 02: 브랜드 리뷰 서비스
```
역할: 기관 브랜드 정보 업데이트 및 RAG 갱신
입력: 브랜드 가이드 변경사항
출력: 갱신된 RAG 임베딩
의존성: db, rag
```

### 모듈 03: 캠페인 계획 서비스
```
역할: 마케팅 캠페인 수립 및 스케줄링
입력: 캠페인 목표, 기간, 예산
출력: 월간 콘텐츠 발행 스케줄
의존성: db, ai-agents
```

### 모듈 04: 콘텐츠 생성 서비스
```
역할: 텍스트/이미지/영상 콘텐츠 자동 생성
입력: 주제, 채널, 기관 RAG 컨텍스트
출력: 콘텐츠 초안 (승인 대기 등록)

서브 모듈:
├── text-gen/     ← 네이버블로그/뉴스레터/스레드 텍스트
├── image-gen/    ← 카드뉴스 (Pillow/Canvas)
└── video-gen/    ← 쇼츠 영상 (MoviePy/Remotion)

의존성: db, rag, ai-agents
```

### 모듈 05: 이메일 자동화 서비스
```
역할: 후원자용 정기 뉴스레터 자동 발행
입력: 뉴스레터 콘텐츠, 수신자 목록
출력: 발행 결과 및 성과 데이터
의존성: db, content-gen
```

### 모듈 06: 모금 캠페인 연동 서비스
```
역할: 온라인 모금 플랫폼 연동 및 성과 추적
입력: 캠페인 콘텐츠, 모금 목표
출력: 모금 페이지 링크 + 성과 데이터
연동: 네이버 해피빈, 카카오같이해
의존성: db, analytics
```

### 모듈 07: 댓글/커뮤니티 관리 서비스
```
역할: SNS 댓글 모니터링 및 자동 답변 초안 생성
입력: 플랫폼 웹훅 이벤트
출력: 댓글 감정 분석 + 답변 초안
의존성: db, ai-agents
```

### 모듈 08: 외부 소스 모니터링 서비스
```
역할: 관련 키워드 뉴스/트렌드 수집 및 콘텐츠 기회 제안
입력: 키워드 설정
출력: 트렌드 리포트 + 콘텐츠 아이디어 제안
의존성: db, ai-agents
```

### 모듈 09: 성과 분석 서비스
```
역할: 채널별/캠페인별 성과 수집, 분석, 보고서 생성
입력: 플랫폼 Analytics API 데이터
출력: 성과 대시보드 + 자동 인사이트 보고서

성과 분석 구성:
├── 콘텐츠 성과 (주제/키워드/이미지 스타일별)
├── 발행 패턴 인사이트 (최적 시간대/요일)
├── 채널별 비교 (인스타 vs 네이버 vs 스레드)
└── 도메인 벤치마크 (유사 기관 평균 비교) ← 핵심 차별점

의존성: db, ai-agents
```

## 6. 레이어 3 - AI 오케스트레이션

### 6.1 에이전트 구조

```
┌─────────────────────────────────────────┐
│         오케스트레이터 (전략 두뇌)         │
│  모델: GPT-4o / Claude Opus              │
│  메모리: memory.md (기관별)               │
│  역할: 판단/위임/유저 소통                 │
└──────┬──────────────────────────────────┘
       │
       ├──→ 콘텐츠 에이전트 (경량 모델)
       │      ├── 텍스트 서브에이전트
       │      ├── 이미지 서브에이전트
       │      └── 영상 서브에이전트
       │
       ├──→ 분석 에이전트 (경량 모델)
       │      ├── 성과 데이터 쿼리
       │      └── 인사이트/보고서 생성
       │
       ├──→ 모니터링 에이전트 (경량 모델)
       │      ├── 댓글 감정 분석
       │      └── 외부 키워드 감시
       │
       └──→ 스케줄러 에이전트 (경량 모델)
              ├── 월간 캠페인 계획
              └── 일별 발행 스케줄 관리
```

### 6.2 LangGraph State 구조

```python
class MarketingState(TypedDict):
    org_id: str                  # 기관 식별자
    current_campaign: dict       # 현재 진행 캠페인
    pending_contents: list       # 승인 대기 콘텐츠
    recent_performance: dict     # 최근 성과
    user_message: str            # 유저 입력
    agent_decision: str          # 에이전트 판단
    priority: str                # URGENT / NORMAL / SCHEDULED
```

### 6.3 에이전트 흐름 예시

```
유저: "이번 주 콘텐츠 계획 잡아줘"
        ↓
오케스트레이터 → memory.md 로드 + 현재 캠페인 확인
        ↓
분석 에이전트 → 지난주 성과 분석
        ↓
오케스트레이터 → 채널 우선순위 결정
        ↓
콘텐츠 에이전트 → RAG 기반 초안 생성
        ↓
스케줄러 에이전트 → 발행 일정 결정
        ↓
대시보드 승인 대기 등록 + 텔레그램 요약 전송
```

                         Supabase Realtime

## 8. 콘텐츠 발행 파이프라인

### 8.1 두 개의 발행 라인

```
스케줄링 라인 (자동)          온디맨드 라인 (요청)
Cron Job 트리거               유저 요청 트리거
      ↓                             ↓
      └──────────┬──────────────────┘
                 ↓
         우선순위 큐
         ├── URGENT (온디맨드 긴급)
         ├── NORMAL (온디맨드 일반)
         └── SCHEDULED (스케줄)
                 ↓
         오케스트레이터
                 ↓
         콘텐츠 에이전트
                 ↓
         승인 대기 등록
                 ↓
         유저 승인/수정
                 ↓
         플랫폼별 발행
```

### 8.2 플랫폼별 발행 방식

```
인스타그램/스레드/페이스북  → Meta API 자동 발행
네이버 블로그              → 대시보드에서 유저 직접 Copy & Paste
                            (네이버 정책상 API 미제공)
유튜브 쇼츠               → YouTube Data API 자동 발행
이메일 뉴스레터            → Mailchimp / Stibee API
```

### 8.3 충돌 방지 로직

```
온디맨드 요청 감지
        ↓
당일 스케줄 확인
        ↓
충돌 감지 시 유저에게 알림:
"오늘 인스타 발행 예정이에요. 추가로 올릴까요, 대체할까요?"
```

## 모듈 조립 원칙

각 서비스 모듈은 다음 인터페이스를 준수해서 독립적으로 개발/교체 가능하게 유지해요.

```typescript
interface ServiceModule {
  name: string
  version: string
  dependencies: string[]      // 의존하는 패키지
  
  execute(input: ModuleInput): Promise<ModuleOutput>
  validate(input: ModuleInput): boolean
  getStatus(): ModuleStatus
}
```

이 구조를 지키면 Phase별로 모듈을 독립적으로 개발하고 조립할 수 있어요.
